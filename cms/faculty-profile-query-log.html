Building faculty-profile-query.vm

A walkthrough of how the Faculty Page sitemap export was split off, scoped to one metadata set, wired to the attached Faculty Profile Block, and flattened into a single-level XML feed.

01 The ask

The existing sitemap-query.vm walks four metadata sets (Page, Faculty Page, Blog Category, Blog) and emits a flat sitemap <url> entry for each. The new requirement: produce a feed scoped to only the Faculty Page set, and enrich each entry with data Cascade doesn't put in a sitemap — the page's displayName, plus fields pulled from the Faculty Profile Block attached to each page.

Before writing anything, we needed the block's Data Definition and the page's Data Definition, since the original template never touches structured data — only path/link/lastPublishedOn off the query result.

03 How it evolved

Revision 1

Scope to Faculty Page, attach displayName + block data

Replaced the 4-set #foreach with a single query filtered by byMetadataSet("Faculty Page"), added .preloadStructuredData() so each result's structured data is fetched in the same pass instead of a round-trip per page, and dereferenced the page's block field to read the attached Faculty Profile Block.

preloadStructuredData() byMetadataSet("Faculty Page") getStructuredDataNode("block").asset

Revision 2

Trim fields, add publish status

Dropped everything from the block except title and primaryDepartment. Added a published / unpublished flag — there's no direct boolean for this on a query result, so it's inferred from whether lastPublishedOn is set. Note this is distinct from the query's publishableOnly(true) filter, which only means the asset is allowed to publish, not that it has been.

lastPublishedOn → published/unpublished

Revision 3

Flatten the output

The block fields had been nested under a <facultyProfileBlock> wrapper. Final pass pulls title and primaryDepartment up to the same level as displayName and status, and the repeating title field (1–5 values) is concatenated into a single comma-separated string instead of a nested list.

04 Final output shape

One <facultyProfile> entry — fully flat

<facultyProfile>
    <loc>https://example.utsa.edu/path/to/page.html</loc>
    <displayName><![CDATA[Jane Doe]]></displayName>
    <status>published</status>
    <lastmod>2026-06-13</lastmod>
    <title><![CDATA[Associate Professor, Graduate Advisor]]></title>
    <primaryDepartment><![CDATA[Department of Example]]></primaryDepartment>
</facultyProfile>

06 Open items before shipping

  • Verify on a live page $page.lastPublishedOn is treated as a string and tested with .isEmpty(). If this Cascade instance returns a Date object instead, that call will throw — swap to $_PropertyTool.isNull($rawPubDate) if so.
  • Verify on a live page .preloadStructuredData() is called with no arguments per the public docs. Some Cascade versions take a boolean — confirm against this instance before relying on it at scale.
  • Engine detail The title-join logic uses $velocityCount (Velocity's built-in 1-based loop counter) to decide where to insert ", ". If this Cascade instance's Velocity config exposes $foreach.count instead, swap it in.
  • Product decision Pages with no Faculty Profile Block attached still produce an entry, with title and primaryDepartment left empty, rather than being skipped. Confirm that's the desired behavior versus omitting them outright.
